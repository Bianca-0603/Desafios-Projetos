No Git, os arquivos podem ter os estados:

  • Unmodified (Não Modificado)
  • Modified (Modificado)
  • Staged (Preparado)
  • Untracked (Não Rastreado).

    • Untracked é para arquivos novos que o Git ainda não conhece; Modified indica que um arquivo rastreado foi alterado; Staged é quando você prepara a alteração para o próximo commit; e Unmodified significa que o arquivo foi salvo no banco de dados do Git sem alterações pendentes.

    Os Estados dos Arquivos no Git Untracked (Não Rastreado): Este é o estado inicial de qualquer arquivo que você adicionou ao seu diretório de trabalho, mas que o Git ainda não está monitorando.O Git não tem nenhuma informação sobre ele, então ele não faz parte do histórico do repositório.

    Unmodified (Não Modificado): Este estado é para arquivos que o Git está rastreando, mas que não sofreram nenhuma alteração desde o último commit. Eles existem no banco de dados do Git e não precisam ser adicionados novamente à área de preparação.

    Modified (Modificado): Você altera um arquivo que já está sendo rastreado, e ele passa para o estado Modified, indicando que há alterações no seu diretório de trabalho. Para que essas modificações sejam salvas em um commit, você precisa adicioná-las à área de preparação.

    Staged (Preparado): Após usar o comando git add, um arquivo modificado é movido para a área de staging. Ele está pronto para ser incluído no próximo commit, que será uma nova "foto" do seu projeto registrada no banco de dados do Git.

    O Ciclo de Vida dos Estados:

    Adicionar um Arquivo:

    Um arquivo novo é Untracked.

    Rastrear o Arquivo: Você usa git add e ele se torna Staged.

    Fazer um Commit: O Git cria uma nova versão do arquivo e ele volta ao estado Unmodified no banco de dados.

    Modificar o Arquivo: Se você editar um arquivo Unmodified, ele se torna Modified.

    Repetir o Ciclo: Você pode adicioná-lo novamente à área de Staged usando git add e criar um novo commit.

    CLONANDO REPOSITÓRIOS

    Para clonar um repositório, você pode usar o comando git clone seguido da URL do repositório que deseja copiar.

    Clonagem significa que outra pessoa pode pegar seu código ou projeto e executar o seu programa e a partir dele fazer contribuições e alterações. O clone fica no diretório local da máquina que executou o comando. Para enviar as alterações de volta, você precisaria fazer um push para o repositório remoto.

    Fetch Origin: O comando git fetch origin é usado para baixar as alterações do repositório remoto, para seu repositório local, mas não as mescla automaticamente com a sua branch local. Ele verifica se há novas alterações no repositório remoto e as traz para o seu repositório local, permitindo que você as revise antes de mesclá-las. Verifica se precisa dar um push (mandar / empurrar para o servidor), ou pull ( coisas do repositório remoto).

    GIT IGNORE - SERVE PARA IGNORAR ARQUIVOS DESNECESSÁRIOS DENTRO DE UMA PASTA OU PROJETO. ELE NÃO VAI NEM VERSIONAR LOCALMENTE NEM ENVIAR PARA O GITHUB QUANDO FIZERMOS UM PUSH.

    Nós não temos autorização de copiar o repositório de ninguém e colocar lá de volta. Mas pode sugerir alterações criando ISSUES.
    ISSUES são uma forma de comunicação entre os colaboradores de um projeto, permitindo que você descreva problemas, solicite melhorias ou faça perguntas sobre o código.

    ISSUES são utilizadas para quando se descobre problema(s) que não temos idéia de como resolver, ou não temos acesso para resolver e precisamos de ajuda.

    PULL REQUESTS (PRs) são uma forma de propor alterações em um repositório. Quando você faz um PR, está pedindo para que suas alterações sejam revisadas e se aprovadas, mescladas ao branch principal do projeto.
